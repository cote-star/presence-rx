# Hero Feature Organization

This note explains the recommended order for the README hero features and the alternate structures worth considering for demos or landing pages.

## Recommended Order

1. Gap analysis
2. Strategic action
3. Channel activation
4. Claim simulation
5. Supporting surfaces

This order is the strongest product architecture because it follows the marketer's decision path. First the user needs to know what is wrong. Then they need to know what type of intervention fits the issue. Then they need to know where to act. Only after that should the system decide what language is safe to publish.

## Why Gap Analysis Comes First

Gap analysis is the truth layer. It establishes that Presence Rx is not inventing a strategy from a generic prompt. It starts from Peec visibility, topic, competitor, engine, and source signals.

For Nothing Phone, the important gap is not simply low visibility. The stronger diagnosis is:

- Smartphone Design is an owned strength.
- Minimalist Hardware is a perception gap because Apple owns the category language.
- Mobile Ecosystem is a discovery gap because the brand wants a cohesive-product-family association, not platform dominance.
- Wireless Audio is an attention gap because products exist but category-level signal is thin.

This creates the base for every later feature. Without gap analysis first, the action layer looks like generic marketing advice.

## Why Strategic Action Comes Second

Strategic action is the translation layer. It converts the gap type into the appropriate intervention class:

| Gap Type | Intervention |
| --- | --- |
| Perception | Positioning, messaging, and explicit trait-language proof |
| Discovery | Canonical pages, structured data, source-of-truth pages, and retrieval monitoring |
| Attention | Sustained content, earned coverage, creator proof, and community signal |

This is where Presence Rx becomes more than a dashboard. It explains why different blind spots require different fixes.

## Why Channel Activation Comes Third

Channel activation operationalizes the strategy. Once the issue and intervention are known, the system can map the work to surfaces:

- Owned content for discovery and perception repair.
- Earned / editorial for perception and attention repair.
- Creator / UGC for attention and discovery.
- Review and comparison surfaces for AI-answer citation behavior.
- Peec monitoring prompts and tags for the measurement loop.

This should sit after strategic action because channel advice depends on the issue type. Leading with channels would make the recommendation feel like a media plan before the diagnosis is proven.

## Why Claim Simulation Comes Fourth

Claim simulation is the guardrail and the strongest demo moment, but it should not be the first feature in the README.

If claim blocking leads the story, Presence Rx can look like a claim-compliance tool. If it follows gap analysis, action, and activation, it becomes the final safety layer in a full decision workflow:

```text
diagnose the gap -> choose the fix -> activate the right channels -> publish only earned claims
```

This makes "the system can say no" more credible because the no is grounded in the same evidence chain as the recommendation.

## Supporting Surfaces Come After

Everything else should support the four-feature spine:

| Surface | Role |
| --- | --- |
| Evidence Ledger | Trust and auditability |
| Presence Verdict | Executive summary of the diagnosis |
| Action Brief | Operational output by gap type |
| Activation Brief | Concrete content, channel, and audience actions |
| Analytics | Visual inspection of priorities and competitors |
| Export Hub | Reuse in reports, demos, and handoff |
| Future Directions | Clearly labeled preview of modeled brand-lift and allocation planning |

These are important, but they should not compete with the hero workflow.

## Alternate Orders Considered

### Option A: Claim Simulation First

Best for a short live demo where the room needs an immediate hook.

Use this order:

1. Claim simulation
2. Gap analysis
3. Strategic action
4. Channel activation
5. Evidence ledger

Tradeoff: it creates a memorable "it can say no" moment, but it can make the product sound narrower than it is.

### Option B: Evidence First

Best for a technical review where trust boundaries matter more than product flow.

Use this order:

1. Source-of-record architecture
2. Gap analysis
3. Evidence ledger
4. Strategic action
5. Claim simulation

Tradeoff: it is rigorous, but it delays the user pain and can feel more like infrastructure than a marketer-facing workflow.

### Option C: Channel Activation First

Best only if the audience already understands the diagnosis and wants the growth plan.

Use this order:

1. Channel activation
2. Strategic action
3. Gap analysis
4. Claim simulation
5. Future directions

Tradeoff: it may look like a generic media recommendation unless the gap taxonomy has already been established.

## Final Recommendation

For the README, use:

```text
Gap Analysis -> Strategic Action -> Channel Activation -> Claim Simulation -> Everything Else
```

For a two-minute demo, consider opening with the claim simulator for 10 seconds, then immediately rewind to the evidence chain:

```text
"Here is the no. Now here is why the system knows to say no."
```

That preserves the demo hook without weakening the product architecture.
